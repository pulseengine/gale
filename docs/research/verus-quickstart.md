# Running Verus on Gale — quick-start

How to run Verus on gale, end-to-end. Covers:

- The verified surface (which files are checked, which are trusted)
- The canonical Bazel target
- The standalone per-module path used for benchmarking (e.g. proof
  synthesis tools, soundness sweeps)
- The `--no-cheating` workflow for maximalist soundness benchmarks

This doc was written in response to a benchmarking enquiry from
[`microsoft/verus-proof-synthesis#45`](https://github.com/microsoft/verus-proof-synthesis/issues/45)
and kept as a stable reference.

> **Project state at the time of writing:** `main` at `13a6ea6`
> (after PRs #31 / #32 / #33). `bazel test //:verus_test` is green;
> the standalone `verus` invocation below also runs end-to-end and
> reports `1046 verified, 0 errors`. Verus toolchain pinned by Bazel:
> `0.2026.02.15.61aa1bf` (Rust 1.93.0). Bazel pinned by
> `.bazelversion` to `9.1.0`. All quoted file/line locations refer to
> this state — line numbers may drift; trust the file content, not
> the line.

## TL;DR

```bash
git clone https://github.com/pulseengine/gale
cd gale
bazel test //:verus_test --test_output=all
```

That single Bazel target verifies every Verus module (51 source
files, 1046 properties as of `13a6ea6` — see Question 1 for the
exact set; the count grows as new specs are added). `rules_verus`
downloads the Verus binary itself; Bazelisk reads `.bazelversion`
(pinned to 9.1.0) and pulls the right Bazel. The only system
prerequisite beyond Bazel + Rust is **Nix** (for an unrelated
toolchain dependency — see "Why Nix?" below).

To invoke Verus directly per-file (no Bazel), see "Running Verus
standalone (per-file)" below — this is the path used for benchmarking
and proof-synthesis work.

## The verified surface — `src/*.rs`

Every Verus-annotated source file lives in `src/`. The authoritative
list is the `VERUS_SRCS` array in `BUILD.bazel` (search for
`VERUS_SRCS =`). At the time of writing, that's **51 files**:

- 39 kernel-feature modules (sem, mutex, msgq, sched, pipe, …)
- `lib.rs` (crate root, `verus!` macro plumbing)
- `zms` (Zephyr Memory Storage)
- 10 helper modules (`cpu_mask`, `ipi`, `mpu`, `mmu`, `systick`,
  `net_buf`, `ipc`, `cbprintf`, `usage`, `spinlock_validate`)

The README says "39 modules" — that count refers to kernel-feature
modules. It does not include `lib.rs`, `zms`, or the 10 helpers. They
*are* still verified by the same Bazel target.

### Two important warnings about scope

**(a) `Cargo.toml [lib]` points at `plain/src/`, not `src/`.** If you
run `cargo check`, open the project in an IDE, or otherwise rely on
Cargo, you will be reading the *Verus-stripped* code, not the verified
code. The split is:

| Tree | What it is | Tools that consume it |
|------|------------|------------------------|
| `src/*.rs` | Single source of truth, Verus-annotated. | Verus (via Bazel) |
| `plain/src/*.rs` | Auto-generated by `verus-strip`, no Verus annotations. | `cargo test`, Kani BMC, clippy, Rocq translation |

For verification work, look at `src/`. `plain/` is derived; never
edit it.

**(b) Some files use `external_body` — these are *trusted*, not
verified.** A `#[verifier::external_body]` function tells Verus to
trust the Rust body without proving anything about it; only the
`requires`/`ensures` contract is enforced at call sites. For
proof-synthesis benchmarking that distinction matters. As of writing,
9 of the 51 files use `external_body`, with 133 occurrences total
(plus 2 `assume_specification` calls — see
[`docs/safety/verification-honesty.md`](../safety/verification-honesty.md)
for the per-module breakdown and what each trusted unit shims):

| File | `external_body` count |
|------|----:|
| `src/net_buf.rs` | 36 |
| `src/pm.rs` | 22 |
| `src/ipc.rs` | 22 |
| `src/mmu.rs` | 20 |
| `src/usage.rs` | 18 |
| `src/thread_lifecycle.rs` | 5 |
| `src/sched.rs` | 4 |
| `src/poll.rs` | 3 |
| `src/mpu.rs` | 3 |

Reproduce: `grep -c external_body src/*.rs | grep -v ':0$'`. Both
`assume_specification` calls live in `src/sched.rs` (in addition to
its 4 `external_body` instances above).

The README's headline "805 properties verified, 0 errors" counts the
`requires`/`ensures` clauses Verus actually discharged via Z3 at the
time the README was last updated. The current count (as of
`13a6ea6`) is **1046 verified, 0 errors** — the README will lag the
true number whenever specs are added. Either way, the count does
*not* subtract trusted bodies. To filter to "fully Verus-discharged"
code, the safest core is the synchronization
primitives (sem, mutex, condvar, futex, atomic, spinlock), msgq /
stack / fifo / lifo / queue, and the timer family — none of those use
`external_body`.

**(c) Other directories that are *not* verified:**

- `tools/verus-strip/` — the stripper itself; not part of the kernel.
- `ffi/` — C-interop shims; tested with Kani BMC, not Verus.
- `tests/`, `benches/`, `fuzz/` — runtime tests, not verified.
- `proofs/` — Rocq (`*.v`) and Lean (`lean/*.lean`) proofs. These
  reason about *abstract* mathematical models, not the Rust code
  directly. See `docs/safety/verification-honesty.md` for the full
  story on what each track actually proves.

## Running Verus on the whole project

### Canonical command (uses Bazel)

```bash
bazel test //:verus_test --test_output=all
```

That target is defined in `BUILD.bazel` (search for `name =
"verus_test"`):

```python
verus_test(
    name = "verus_test",
    srcs = VERUS_SRCS,
    crate_name = "gale",
)
```

It runs Verus over all 51 files together as a single crate
(`crate_name = "gale"`). Cross-module specs are resolved as one build
— that matters because some `proof fn`s and spec functions are re-used
across files (e.g. `wait_queue` specs are imported by `sem`, `mutex`,
etc.).

`--test_output=all` is what gets you the per-module property counts on
stdout. With the default `--test_output=errors` you only get
`PASSED`/`FAILED`.

CI proof of correctness:
`.github/workflows/formal-verification.yml`, the `verus` job. This is
the same command CI runs — a green CI run on `main` means the command
succeeds.

### Prerequisites

| Tool | Why | How to install |
|------|-----|----------------|
| **Bazelisk** | Drives Bazel | `brew install bazelisk` / `npm i -g @bazel/bazelisk` / [GH releases](https://github.com/bazelbuild/bazelisk/releases) |
| **Nix** | Required for toolchain resolution (see below) | [nixos.org/download](https://nixos.org/download) |
| **Rust toolchain** | `rust-toolchain.toml` pins stable | `rustup` (it auto-installs the channel from `rust-toolchain.toml`) |

You do **not** need to install Verus, Z3, Rocq, or Lean separately —
`rules_verus` and friends pull them in.

`.bazelversion` pins Bazel to **9.1.0**; Bazelisk will fetch it
automatically. (Earlier versions of this doc mentioned no pin
existed — that was true before PR #32 and is now fixed.)

### Why Nix? (you don't need `nix develop`)

You need `nix` on `PATH`, but you do **not** need `nix develop` (and
there is no `flake.nix` in the repo). The reason:

- `rules_verus` downloads Verus by itself — Verus is not the reason
  Nix is needed.
- `rules_rocq_rust` (declared in `MODULE.bazel`) pulls
  `rules_nixpkgs_core` for Rocq toolchain resolution.
- Even when running *only* the Verus target, Bazel resolves the full
  module graph, so `nix` has to be on `PATH`.

The CI workflow does this directly: it installs Nix via
`cachix/install-nix-action`, then runs `bazel test //:verus_test` —
no `nix develop` wrapper. So: install Nix system-wide, then run
`bazel test //:verus_test`.

(Earlier versions of `AGENTS.md` and this doc mentioned a `nix
develop --command` prefix; that was incorrect and was removed in PR
#32.)

### First-run cost

The first `bazel test //:verus_test` fetches `rules_verus` (Verus +
Z3), `rules_rocq_rust` (Rust + Rocq via Nix), and transitively
`rules_lean` + Mathlib. We have not measured the cold-cache time
ourselves; expect it to be slow on first run and fast on subsequent
runs.

### Output

A successful run ends with the Verus stdout line:

```
verification results:: 1046 verified, 0 errors
```

(measured on `main @ 13a6ea6`). On failure, `bazel test` exits
non-zero and Verus prints the unsatisfied proof obligation with
source location.

### Running a single module via Bazel

`rules_verus`'s rules in this repo only expose the all-or-nothing
target. To verify just one module under Bazel, the easiest path is to
edit `BUILD.bazel`, narrow `VERUS_SRCS` to your subset, and re-run.
For per-module work, the standalone path below is cleaner.

## Running Verus standalone (per-file, no Bazel)

This is the path I'd recommend for `verus-proof-synthesis` work — it
gives you a single Verus invocation per module and bypasses the entire
Bazel/Nix stack.

### Get a Verus binary

Pin to the version Gale's CI uses:

```
Verus 0.2026.02.15.61aa1bf
Toolchain: Rust 1.93.0 (bundled)
```

Easiest options:

```bash
# Option A: download a prebuilt release matching the version
# https://github.com/verus-lang/verus/releases
# Look for the tag '0.2026.02.15' or commit '61aa1bf'.

# Option B: reuse the binary Bazel already downloaded
# After `bazel build //:verus_test` once, find it via:
#   find $(bazel info output_base) -name verus -type f -path '*verus_toolchains*'
# (on macOS arm64 the path looks like:
#   <bazel cache>/external/+verus+verus_toolchains_aarch64_apple_darwin/verus)
# That directory also has `rust_verify`, `z3`, `libvstd.rlib`,
# `libverus_builtin.rlib`, `libverus_builtin_macros.dylib`,
# and a bundled `rust_sysroot/`. You can copy the whole directory
# elsewhere and call `verus` from it.

# Option C: build from source (slow)
git clone https://github.com/verus-lang/verus
cd verus/source
./tools/get-z3.sh
source ../tools/activate
vargo build --release
```

### Invoke Verus on the whole crate

The exact invocation Bazel uses (see
`rules_verus/verus/private/verus.bzl`, lines ~165–175 — function
`_verus_verify_impl`) is:

```bash
rust_verify \
  --edition=2021 \
  --crate-type lib \
  --sysroot "$SYSROOT" \
  --extern builtin="$TOOLCHAIN/libverus_builtin.rlib" \
  --extern builtin_macros="$TOOLCHAIN/libverus_builtin_macros.dylib" \
  --extern vstd="$TOOLCHAIN/libvstd.rlib" \
  --crate-name gale \
  src/lib.rs
```

with `DYLD_LIBRARY_PATH` (macOS) or `LD_LIBRARY_PATH` (Linux) set to
include the toolchain dir. Two surprises worth flagging:

1. **Edition is 2021, not 2024.** `Cargo.toml` says edition 2024, but
   that's for the `cargo` build of the *stripped* code. Verus itself
   runs the source through edition 2021. The `verus!` macro handles
   the differences.
2. **The actual binary is `rust_verify`.** `verus` (the wrapper script
   in the toolchain dir) is a thin shell over `rust_verify` that sets
   the library paths. Either one works; using `verus` is friendlier.

The wrapper-friendly version:

```bash
verus --crate-type lib --crate-name gale src/lib.rs
```

(The `verus` wrapper applies `--edition=2021`, `--sysroot`, and the
`--extern` flags for you.)

### Useful flags for benchmarking

These are all real Verus flags (verified against the bundled binary):

| Flag | What it does |
|------|--------------|
| `--time` | Print total verification time |
| `--time-expanded` | Per-module time breakdown |
| `--profile-all` | Always collect Z3 prover performance data (per-query) |
| `--rlimit FLOAT` | SMT resource limit, roughly seconds. Useful for reproducibility — pin it. |
| `--verify-module foo` or `--verify-module foo::bar` | Verify only the named module(s); can be repeated. **Caveat:** Verus still parses the whole crate from `src/lib.rs` before isolating to `foo`, so a parse error anywhere (e.g., another module's `ensures` clause) poisons the run even when you only target `foo`. If a fresh-checkout `--verify-module` invocation fails on a module you didn't ask about, look for cross-module parse breakage first. |
| `--verify-root` | Verify only the crate root, skip submodules. |
| `--no-cheating` | Reject `assume`, `admit`, `external_body`, and `assume_specification` — gives you a stricter proof obligation set. Today gale does NOT pass `--no-cheating` at the crate level because we have 133 `external_body` instances + 2 `assume_specification` calls. **Important caveat:** `--no-cheating` errors at the whole-crate parse/translation phase, so combining it with `--verify-module sem` does *not* skip the rejection — Verus still scans `poll.rs`, `net_buf.rs`, etc. and fails. To benchmark a single clean module under `--no-cheating`, build a minimal crate root that only imports the target module and its deps (see "Per-module standalone bench" below). |

So a per-module benchmarking invocation (without `--no-cheating`) looks like:

```bash
verus --crate-type lib --crate-name gale \
  --verify-module sem \
  --time-expanded \
  --rlimit 30 \
  src/lib.rs
```

This produces per-module timing output (verify-time, air-time, smt-time)
broken down per function — exactly what you want for benchmark
attribution. Wall-clock for the `sem` module on an M-series Mac:
~2 seconds.

### Per-module standalone bench (with `--no-cheating`)

Because `--no-cheating` triggers at parse time across the whole
crate, you cannot use it on `src/lib.rs` directly — the 9 modules
that contain `external_body` will fail before your target module is
ever reached. Two workarounds:

**Option 1 — Strip the trusted modules from `lib.rs`.** Make a copy
of `src/lib.rs` (e.g. `src/lib_strict.rs`) and remove the `pub mod`
lines for the 9 files in the `external_body` table. A one-liner:

```bash
sed -E '/^pub mod (ipc|mmu|mpu|net_buf|poll|pm|sched|thread_lifecycle|usage);$/d' \
  src/lib.rs > src/lib_strict.rs
```

That removes the same 9 modules listed in the `external_body` table
above (`lib_strict.rs` is now the crate root, replacing `lib.rs`; it
declares 41 submodules instead of 50). Then:

```bash
# Per-module strict bench
verus --crate-type lib --crate-name gale \
  --no-cheating \
  --verify-module sem \
  --time-expanded \
  src/lib_strict.rs

# Whole clean-core strict run
verus --crate-type lib --crate-name gale \
  --no-cheating \
  src/lib_strict.rs
# → verification results:: 851 verified, 0 errors
```

So the **clean-core benchmark surface is 851 properties** (out of
1046 total). The other 195 properties live in modules that depend on
trusted bodies — they verify under default Verus but not under
`--no-cheating`.

Per-module wall-clock for `sem` on an M-series Mac: ~150 ms
verification + ~2 s rust-time overhead.

**Option 2 — Per-module crate.** Synthesize a minimal `lib.rs` that
imports only the target module and its transitive deps. More work,
but isolates the bench from cross-module noise. The dependency graph
is shallow: most modules depend on `error`, `priority`, `thread`,
`wait_queue`, and a few primitives.

Either way, the canonical "how many properties can a synthesizer
recover for module X under maximal strictness" question is answered
by Option 1 with `--no-cheating --verify-module X`.

### Verus annotations to look for

Open `src/sem.rs` (good first read — well documented at the top) and
you'll see:

```rust
verus! {
    impl Semaphore {
        pub fn give(&mut self)
            requires
                old(self).count <= old(self).limit,
            ensures
                self.count <= self.limit,
                self.count == old(self).count + 1
                    || self.count == old(self).count,
        { ... }
    }
}
```

The `requires`/`ensures` clauses are the proof obligations Verus
discharges via Z3. The top docblock of `src/sem.rs` lists the proof
properties (P1–P10) and how they map to upstream Zephyr functions.
That pattern repeats across modules — `src/sem.rs:1-30` is the
clearest orientation point.

## Quick honesty pointer

`docs/safety/verification-honesty.md` is the document you should read
before benchmarking. Summary of what it says, specifically about
Verus:

- All 51 files in `VERUS_SRCS` *do* pass Verus locally (1046 properties as of `13a6ea6`).
- Verus runs in CI on `push`/`pull_request` to `main` (workflow:
  `formal-verification.yml`).
- It does *not* run on every commit to non-`main` branches — only when
  the branch is opened as a PR, or on the weekly cron.
- `external_body` blocks count as trusted code (see Question 1
  warning (b) above).
- Rocq and Lean tracks reason about abstract models, not the Rust
  code, so for a "what does Verus actually verify" benchmark you
  should ignore `proofs/`.

## Things that might trip you up

1. **The 39 vs. 51 file count** — README says 39 verified modules,
   `BUILD.bazel`'s `VERUS_SRCS` lists 51 files. Both are right; they
   count different things (see Question 1).
2. **`Cargo.toml [lib]` points at `plain/src/`, not `src/`** — IDE/cargo
   reads stripped code by default. For verified surface, look at
   `src/` directly.
3. **Edition mismatch:** `Cargo.toml` says 2024, but Verus runs
   edition 2021. The `verus!` macro spans the gap. If you run
   `rust_verify` standalone, pass `--edition=2021`.
4. **`external_body` is trusted, not verified** — 133 instances
   across 9 files (plus 2 `assume_specification`). See Question 1
   warning (b) and `docs/safety/verification-honesty.md`.
5. **Property count drifts** — README quotes 805; current count
   (`bazel test //:verus_test --test_output=all` or the standalone
   `verus` invocation) is 1046. The README will lag whenever specs
   are added.
